## 1. 产品概述

**JoyServe** 是面向东南亚跨境电商卖家的 Agent-Native AI 客服 SaaS 管理平台。

区别于传统客服工具，JoyServe 以 **AI Agent** 作为核心产品形态，让卖家像培训新员工一样配置 Agent，Agent 自主处理客服工作，人工仅需在关键节点介入。

**产品核心理念：**

> 像培训一个新员工一样配置 Agent，它上岗后自己干活，你只管看结果。

**传统客服 SaaS 思路 vs JoyServe 思路：**

| 维度 | 传统思路 | JoyServe 思路 |
|------|---------|---------------|
| 交互方式 | 配置规则、话术树、关键词 | 描述业务意图，选择技能 |
| 知识管理 | 手动维护 Q&A 问答对 | 上传文档 + 商品数据，AI 自动提炼 |
| 上线门槛 | 1-2 周配置调试 | 5 分钟冷启动 |
| 人工协作 | AI 失败才转人工 | AI 给人工提供摘要和建议 |
| 效果反馈 | 会话量、响应时间 | 场景覆盖率、ROI、诊断建议 |
| 高级能力 | 几乎没有 | Manager Agent、A/B测试、演练场 |

## 2. 目标用户与核心痛点

### 用户画像

**主要用户：** 东南亚跨境电商中小卖家

**典型场景：**
- 店铺同时开在 Shopee、Lazada、TikTok Shop 多个平台
- 买家使用越南语、泰语、印尼语、马来语等多种语言
- 旺季（双十一、年货节）会话量暴增，人力严重不足
- 客服招募成本高，流动性大，培训周期长

**核心痛点：**

| 痛点 | 具体表现 |
|------|---------|
| 多平台难管理 | 每个平台单独处理，消息经常遗漏 |
| 语言壁垒 | 买家语言多样，客服招聘困难 |
| 重复工作量大 | 80% 的问题是相似的订单、物流查询 |
| 旺季人力不足 | 高峰期响应慢，差评率上升 |
| 转化率偏低 | 咨询转化率低，导购能力弱 |
| 差评处理滞后 | 没有时间及时回复差评 |

### 用户角色划分

```
店铺老板 / 运营负责人
  └── 关注：ROI、AI 覆盖率、整体数据
  └── 使用场景：首页大盘、数据分析、Agent 配置

客服主管
  └── 关注：团队效率、接管分配、质量监控
  └── 使用场景：客服工作台、绩效数据

一线客服
  └── 关注：今天需要我处理哪些、如何快速回复
  └── 使用场景：会话列表、待办事项
```

## 3. 产品定位与核心价值主张

### 价值主张（三层递进）

**第一层：省钱省人力（基础价值）**
> AI 自动处理 80% 的重复性咨询，客服团队专注高价值工作

**第二层：提升买家体验（体验价值）**
> 7×24 小时多语言响应，平均响应时间 < 2 秒，买家满意度提升

**第三层：从成本中心变利润中心（战略价值）**
> AI 主动导购、催付、催评，直接驱动 GMV 增长

### 核心差异化定位

| 竞品定位 | JoyServe 定位 |
|---------|---------------|
| 客服工具（回答问题） | AI 店员（经营买家关系） |
| 处理单次对话 | 管理买家全生命周期 |
| 被动响应 | 主动经营助手 |
| 降低客服成本 | 同时降低成本 + 增加收入 |

## 5. 核心模块：AI 客服

### 5.1 冷启动引导流程

新用户首次进入"AI 客服"模块，通过三步引导 **5 分钟内上线第一个 Agent**。

**设计原则：** 所有问题是业务问题，不是技术问题。用户不需要了解 AI 模型、Prompt、API 等概念。

#### Step 1：选择首个场景（约 30 秒）

```
你想先解决哪个最头疼的问题？

  📦  买家一直问订单/物流        [最受欢迎]
  💸  退款退货处理太耗时
  ⭐  差评太多没时间回复
  👋  新买家没人及时回复
  ⏰  催付催评全靠手动
```

**交互说明：**
- 单选，引导用户聚焦最迫切的场景
- 标注"最受欢迎"降低选择焦虑
- 选择后 Agent 自动预载对应 Skill 组合

#### Step 2：告诉 Agent 你的店铺情况（约 1 分钟）

```
你卖什么类目？
  [女装] [3C数码] [美妆] [食品] [家居] [运动] [其他]

主要服务哪个平台？（可多选）
  [Shopee ✓] [Lazada] [TikTok Shop] [Tokopedia]

你希望 Agent 的回复风格？
  ○ 亲切活泼    ● 专业礼貌    ○ 简洁高效

主要服务语言？
  ● 自动识别买家语言（推荐）
  ○ 指定语言
```

**交互说明：**
- 全部使用选择题，不要求用户填写任何文本
- "自动识别买家语言"为默认选项，降低多语言配置门槛

#### Step 3：预览并上线（约 30 秒）

```
你的第一个 Agent「智能接待员」已就绪

  [预览对话效果 →]      [立即上线]

✓ 已自动配置：
  • 订单查询（接入 Shopee 订单系统）
  • 物流追踪（支持主流物流商）
  • 欢迎语（根据你的品类自动生成）
  • 7×24h 服务，支持越南语/泰语/印尼语等
```

**交互说明：**
- 展示"已自动帮你做了什么"，让用户感受到价值
- "预览对话效果"入口引导进入演练场
- 上线后可随时修改，降低决策负担

### 5.2 Agent 架构设计

#### 三层架构总览

```
买家消息
    ↓
┌─────────────────────────────────────┐
│           路由层（意图识别）           │
│  判断消息类型，分发给对应子 Agent      │
└─────────────────────────────────────┘
    ↓
┌──────────┬──────────┬──────────┬──────────┐
│ 接待Agent │ 物流Agent │ 退款Agent │ 导购Agent │
│ 欢迎/引导  │ 查询/追踪  │ 处理/审批  │ 推荐/促成  │
└──────────┴──────────┴──────────┴──────────┘
    ↓ 遇到边界情况或需跨 Agent 协作时
┌─────────────────────────────────────┐
│           Manager Agent              │
│                                     │
│  · 跨 Agent 协调（多意图整合）        │
│  · 权限升级（子 Agent 无权处理的决策）  │
│  · 转人工决策（评估是否真的需要人工）   │
│  · 店铺异常熔断（系统性问题应对）      │
└─────────────────────────────────────┘
    ↓ 真正需要人工时
┌─────────────────────────────────────┐
│             人工客服                  │
│  接收：完整对话记录 +                 │
│        Manager 分析报告 +            │
│        建议处理方案                  │
└─────────────────────────────────────┘
```

#### Agent 的本质定义

每个 Agent = **身份（人设）** + **能力（Skill 组合）** + **范围（服务边界）**

```
身份：叫什么名字、什么风格、代表哪个品牌
能力：能处理哪些场景（来自技能市场的 Skill）
范围：服务哪个平台、哪些店铺、什么时段
```

**关键设计原则：**
- Agent 配置全程不涉及大模型选型、Prompt 编写、API Key 等技术参数
- 所有技术参数由 SaaS 平台后台统一管理和优化
- 用户只需用业务语言描述"我要 Agent 干什么"

### 5.3 Manager Agent

#### 设计背景与问题

现有 AI 客服的主要问题：

| 问题 | 表现 | 后果 |
|------|------|------|
| 转人工太随意 | 子 Agent 稍有困难就转人工 | 浪费人力资源 |
| 转人工太生硬 | 突然切换，体验断层 | 买家体验差 |
| 无中间缓冲 | 高峰期人工队列爆满 | 等待时间长，投诉增加 |
| 知识无法沉淀 | 每次遇到新问题都转人工 | 系统无法自我进化 |

**Manager Agent 的角色定位：**

```
子 Agent    =  一线客服专员（处理常规问题）
Manager Agent =  客服主管（有审批权限，协调资源）
人工         =  真正的人类客服经理（处理极端情况）
```

引入 Manager Agent 后，真正需要人工介入的比例预计从 **20% 降至 3-5%**。

#### Manager Agent 的三重角色

**角色一：决策者**

有权限自主处理子 Agent 无权处理的业务决策：

```
退款审批
  ├── 金额 < $50 且符合规则    → 直接审批通过
  ├── 金额 $50-200 有充分凭证  → 审批 + 记录备查
  └── 金额 > $200             → 转人工，附完整分析报告

补偿授权
  ├── 发放 $20 以内优惠券      → 自主决定
  ├── 申请加急发货             → 自主操作
  └── 承诺部分退款（≤30%订单金额）→ 自主处理

信息权限
  ├── 调取买家全部历史订单      → 有权限
  ├── 查询店铺库存、物流后台    → 有权限
  └── 跨店铺查询同一买家记录   → 有权限
```

**角色二：协调者**

编排多个子 Agent 协同解决复杂问题：

```
示例：买家发来一条消息

  "我收到的商品破损了想退款，
   另外我之前买过你们产品，能给我优惠吗？"

涉及多个 Agent：
  → 退款处理 Agent（破损退款流程）
  → 买家画像 Agent（历史购买记录）
  → 营销 Agent（优惠券发放权限）

Manager 的工作：
  1. 拆解消息中的多个意图
  2. 并行调度三个子 Agent
  3. 整合结果，给买家一个连贯流畅的回复
  4. 买家感知不到背后的复杂调度
```

**角色三：守门人**

评估是否真的需要人工，而不是简单地透传转人工请求：

```
子 Agent 请求转人工时，Manager 先评估：

情况 A：子 Agent 遇到了没见过的问题
  → 先在全局知识库搜索
  → 调用其他子 Agent 的能力组合解决
  → 解决不了再转人工，同时记录这个知识盲点

情况 B：买家情绪激动
  → 评估投诉风险等级
  → 低风险：给子 Agent 更高级别话术权限继续
  → 中风险：Manager 亲自接管，提升服务规格
  → 高风险：转人工，Manager 准备完整处理建议

情况 C：买家问了超出授权的退款问题
  → 检查该买家历史记录和 VIP 等级
  → 评估是否可以给予特殊授权
  → 有依据则自主处理，无依据才转人工
```

#### Manager Agent 的特殊能力

**能力一：动态权限调整**

```
场景                     Manager 动作
────────────────────────────────────────────
大促期间                  临时将退款自动审批上限从 $50 提升到 $100
买家是 VIP（消费 > $500）  给该会话临时更高服务权限
店铺有活动                向所有子 Agent 广播活动规则
物流大面积延误            暂停自动审批，统一告知买家情况
```

**能力二：店铺异常感知与熔断**

```
触发条件                          Manager 响应
────────────────────────────────────────────────
单小时退款会话量上涨 > 200%        检测是否是特定 SKU 批量问题
                                  如是，暂停该商品自动审批
                                  通知运营人员

同一问题 1 小时内重复出现 > 20 次   自动查询相关业务状态
                                  发现问题则暂停对应类型回复
                                  主动给受影响买家发送说明

情绪评分跌至历史最低               全面提升服务规格
                                  优先分配人工资源
                                  推送告警给管理员
```

**能力三：人工接管的软着陆**

区别于传统的"硬切换"，Manager 提供平滑过渡：

```
Step 1  Manager 先接管对话（买家感知是"主管介入"）
        "您好，我是这里的客服主管，刚才了解了您的情况，
         让我来帮您处理。"

Step 2  后台同时为人工客服准备完整材料
        ┌──────────────────────────────────────┐
        │  🤖 AI 工作摘要（仅人工客服可见）       │
        │                                      │
        │  转交原因：买家情绪激动（连续3条负面）   │
        │                                      │
        │  对话摘要：                           │
        │  买家反映订单延误，AI 已查询物流显示    │
        │  包裹滞留清关，买家要求退款。           │
        │                                      │
        │  建议处理方式：                       │
        │  ① 先安抚情绪，说明清关原因            │
        │  ② 提供补偿方案（优惠券 or 加急）      │
        │  ③ 如坚持退款，金额 $18.5，在授权内    │
        │                                      │
        │  [使用 AI 起草回复]  [直接输入]        │
        └──────────────────────────────────────┘

Step 3  人工客服接手时，体感像是在继续之前的对话
        买家无需重新介绍自己的问题

Step 4  人工处理完后，Manager 记录
        "这类问题下次如何让 AI 直接处理？"
        自动将处理方案补充到知识库
```

**能力四：跨店铺、跨平台协同**

```
买家在 Shopee 下单，通过 Lazada 店铺客服反馈问题
  → Manager 识别是同一买家（手机号/邮箱匹配）
  → 调取 Shopee 的订单信息
  → 在 Lazada 渠道给出完整解答

买家体验：不用管在哪个平台，一个入口解决所有问题
```

#### Manager Agent 绝对不处理的情况

以下情况直接转人工，不做任何拦截：

- 买家明确强烈要求人工（不得阻拦）
- 涉及法律纠纷（买家提到律师/法院/起诉）
- 平台已启动官方纠纷流程
- 媒体/网红买家威胁曝光
- 疑似欺诈（同一买家多次异常退款）
- 退款金额超过设定的最高授权上限
- 涉及账号封禁、平台违规等重大问题

### 5.5 Agent 配置体系

## 6. 知识库设计

### 6.1 商品数据打通

**核心价值：** 打通商品数据后，Agent 不再只是"回答问题"，而是真正"懂你的货"，从客服工具升级为智能导购。

#### 支持同步的数据类型

| 数据类型 | 来源 | 用途 |
|---------|------|------|
| 商品基础信息 | Shopee/Lazada/TikTok 商品详情 | 回答商品咨询 |
| SKU 与规格 | 平台 SKU 数据 | 精准回答尺码、颜色等问题 |
| 实时库存 | 平台库存系统 | 库存感知的应答策略 |
| 商品价格 | 平台价格数据 | 告知当前价格和活动折扣 |
| 商品图片/视频 | 平台媒体资源 | 图文回复，提升导购体验 |
| 活动规则 | 平台活动系统 | 实时告知活动信息 |
| 物流时效 | 仓库/物流系统 | 准确告知预计到达时间 |

#### 商品数据打通后的新能力

**智能导购场景：**
```
买家：我想买一件适合参加婚礼的裙子，预算200以内

Agent：根据您的需求，为您推荐以下几款：
       [商品卡片 1：实时库存 S/M/L，$18.9，本周9折]
       [商品卡片 2：限量款，仅剩 3 件，$24.5]

       请问您是什么身材，我帮您确认尺码？
```

**库存感知应答策略：**
```
买家：这双鞋还有38码吗？

有货 → 告知库存，顺带提醒："仅剩 2 件，建议尽快"
无货 → 推荐相近款，或提供到货提醒订阅
库存仅 1 件 → "仅剩最后一件，建议立即下单"（制造紧迫感）
```

**价格与活动感知：**
```
买家：你们这个商品最近会打折吗？

有活动 → 告知活动规则，引导下单
即将有活动 → "本周五大促，届时9折，可先加购物车"
参考历史价 → "这款商品当前价格在近期属于较低水平"
```

**买家描述不清楚时的反向识别：**
```
买家：我买的那个东西坏了（未说是哪件商品）

Agent 自动查询该买家最近订单：
"您好，是您上周购买的那款无线耳机出现问题了吗？"

买家确认后直接进入对应处理流程
买家无需提供订单号，Agent 自己猜对了
```

### 6.2 知识录入方式

采用**多渠道录入 + AI 自动提炼**，彻底告别手动维护 Q&A 的繁琐工作。

#### 方式一：上传文档（推荐）

```

  上传文档 → AI 阅读理解 → 提取关键知识点
           → 关联到对应场景 → 支撑 Agent 回答
```

#### 方式四：对话中自动学习

## 8. 智能营销

### 8.1 主动关怀时间线

这是从"被动客服"到"主动经营"的核心功能，整条时间线**全自动运行**：

```
下单后 1 小时    → 确认收到订单，告知预计发货时间
发货后           → 通知快递单号，告知预计时效
包裹即将到达     → 提醒留意收货
确认收货后 3 天  → 询问使用体验，引导好评
好评后           → 感谢 + 推荐关联商品（二次复购）
```

### 8.2 物流异常主动干预

```
系统检测到：买家订单物流 48 小时无更新

传统做法：等买家来投诉
我们的做法：Agent 主动发消息

  "您好，我们注意到您的包裹在清关环节略有延迟，
   预计额外需要 1-2 天。非常抱歉带来不便！
   作为补偿，为您发放一张 $5 优惠券，感谢您的耐心等待。"

效果：把投诉消灭在萌芽状态，还能提升好评率
```

### 8.3 沉默客户唤醒

```
识别条件：上次购买超过 60 天且未再下单的老客户

策略（不是群发优惠券，而是个性化关怀）：
  → 分析上次买了什么、是否遇过问题
  → 推荐相关新品，附带专属优惠
  → 时机选择：下午 2-4 点（开屏率最高时段）

示例消息：
```

## 11. 差异化竞争策略

### 11.1 核心差异化三大命题

**命题一：从"客服工具"到"AI 店员"**
> 竞品帮你回答买家问题
> JoyServe 帮你经营买家关系、推动成交、降低流失

**命题二：从"单次对话"到"全生命周期管理"**
> 竞品处理当下这条消息
> JoyServe 管理买家从第一次咨询到复购的整个旅程

**命题三：从"被动响应"到"主动经营助手"**
> 竞品等买家来问
> JoyServe 帮卖家提前发现问题、主动出击、把客诉消灭在发生前

### 11.2 竞品差距全景

| 维度 | 传统客服 SaaS | JoyServe |
|------|-------------|-----------|
| 配置方式 | 关键词规则、话术树 | 业务意图描述 + Skill 选择 |
| 知识管理 | 手动维护 Q&A | 文档上传 + 商品数据自动同步 |
| 上线门槛 | 1-2周 | 5分钟冷启动 |
| Agent 架构 | 单层 AI | 子 Agent + Manager Agent 双层 |
| 人工协作 | AI 失败才转人工 | AI 给人工摘要和建议，软着陆 |
| 商品能力 | 无 | 实时库存/价格/SKU 感知 |
| 买家记忆 | 无 | 跨会话记忆，持续丰富买家画像 |
| 主动服务 | 无 | 购后关怀时间线全自动运行 |
| 效果反馈 | 会话量、响应时间 | 场景覆盖率、ROI、诊断建议 |
| 多模态 | 纯文字 | 支持图片识别（买家发图处理） |
| 情感智能 | 基础情绪检测 | 四级情绪分级响应策略 |

### 11.3 情感智能分级策略

```
Level 1  轻微不满（语气较急）
  → Agent 适当放缓语气，加一句安抚，不转人工

Level 2  明显不满（出现投诉词汇）
  → Agent 切换高级别道歉话术
  → 自动给予一定权限的补偿（如 $3 优惠券）
  → 后台给客服发预警通知

Level 3  激烈投诉（威胁差评/投诉平台）
  → 立即转人工
  → 转接时附完整情绪分析报告
  → 自动标记该买家为"高风险"

Level 4  潜在法律风险（出现特定词汇）
  → 立即转高级客服
  → 对话内容锁定存档备查
  → 通知管理员
```

### 11.4 多模态理解能力（P1规划）

```
买家发来破损照片
  → Agent 识别破损类型（包装损坏/商品本体损坏）
  → 判断是否符合退款标准
  → 自动启动退款流程 or 要求补充凭证

买家发截图问："这个色号你们有吗？"
  → Agent 识别截图中的商品颜色/款式
  → 在自家商品库中匹配
  → 回复：有/没有，并推荐替代款
```

## 13. 设计原则与规范

### 13.1 产品设计核心原则

**原则一：零配置可用，深度配置可选**
> 新用户 5 分钟内让 Agent 运行起来
> 高级配置（接管规则、情感阈值、跨平台差异化）按需展开
> 不要把复杂度堆在第一屏

**原则二：业务语言，不是技术语言**
> 用户配置的是"我希望 Agent 怎么处理退款"
> 而不是"设置 temperature 参数"或"编写 System Prompt"
> 所有配置项用业务场景描述，不暴露技术细节

**原则三：数据驱动的主动引导**
> 不让用户自己发现问题
> 系统主动告知："你的退款场景解决率偏低，建议优化"
> 每个告警都附带解决方案，不只是报警

**原则四：结果可见，价值可量化**
> 让用户清晰地看到"AI 帮我省了多少钱多少时间"
> 定期推送 ROI 报告，形成正向激励

**原则五：AI 辅助人工，而非替代人工**
> 人工接管时 AI 提供摘要和建议，不是甩手不管
> 让人工客服工作更轻松，而不是让他们担心被取代

### 13.2 界面设计规范

- **设计风格：** 现代、简洁的浅色系（参考 TDesign 规范）
- **主色调：** 品牌蓝 #0052D9
- **功能色：** 成功绿 #00A870，警告橙 #ED7B2F，错误红 #E34D59
- **字体：** 中文 PingFang SC / 微软雅黑，英文 -apple-system
- **最低分辨率：** 1366 × 768（Desktop First）

### 13.3 不在我们平台暴露给用户的技术配置

以下内容由 SaaS 平台后台统一管理，用户界面不展示：

- AI 大模型选型（Claude / GPT 等）
- System Prompt 内容
- API Key 配置
- Temperature、Max Token 等模型参数
- 流式回复（SSE）开关
- 向量数据库配置
- 模型切换策略
